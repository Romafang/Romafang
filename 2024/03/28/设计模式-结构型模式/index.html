结构型模式

适配器模式

适配器模式（Adapter Pattern）是一种结构型设计模式，它允许接口不兼容的类之间进行合作。适配器模式通过创建一个包装类来将一个类的接口转换成客户端期望的另一个接口，使得原本由于接口不兼容而无法在一起工作的类可以一起工作。

适配器模式通常包括以下几个角色：

  1. 目标接口（Target）：客户端期望的接口，适配器模式通过实现这个接口来使得适配器类能够与客户端交互。
  2. 适配器类（Adapter）：包装了被适配者对象，并实现了目标接口。适配器类将客户端的请求委派给被适配者对象。
  3. 被适配者类（Adaptee）：需要被适配的类，它拥有不兼容的接口。

桥接模式

桥接模式（Bridge Pattern）是一种结构型设计模式，它通过将抽象部分与实现部分分离，使它们可以独立地变化。桥接模式的核心思想是将一个大类或一组类分成两个独立的层次结构，从而可以在两个层次之间建立一个桥接，使它们可以独立地变化。

桥接模式主要包括以下几个角色：

  1. 抽象类（Abstraction）：定义了抽象部分的接口，并维护一个指向实现部分对象的引用。
  2. 扩展抽象类（Refined Abstraction）：扩展了抽象类，通常是为了增加更多的功能。
  3. 实现接口（Implementor）：定义了实现部分的接口，该接口不一定和抽象部分完全一致。
  4. 具体实现类（Concrete Implementor）：实现了实现部分的接口，提供具体的功能。

组合模式

组合模式（Composite Pattern）是一种结构型设计模式，它允许将对象组合成树形结构以表示“部分-整体”的层次结构。组合模式通过使得单个对象和组合对象具有一致的接口，使得客户端可以统一地对待单个对象和组合对象，从而简化了客户端代码。

组合模式主要包括以下几个角色：

  1. 组件（Component）：定义了组合中对象的共有接口，可以是抽象类或接口。组件类通常包含管理子组件的方法，例如添加子组件、删除子组件、获取子组件等。
  2. 叶子（Leaf）：表示组合中的叶子节点对象，叶子节点没有子节点。通常是组合的基本对象，它实现了组件接口。
  3. 容器（Composite）：表示组合中的容器节点对象，容器节点包含子节点。容器节点可以有一个或多个子节点，它实现了组件接口，并提供了管理子节点的方法。

装饰模式

装饰模式（Decorator Pattern）是一种结构型设计模式，它允许在不改变对象自身的基础上动态地添加功能。装饰模式通过将对象包装在装饰器对象中，从而实现对对象的功能扩展，同时保持接口的一致性和组合性。

装饰模式主要包括以下几个角色：

  1. 组件（Component）：定义了被装饰的对象的接口，可以是抽象类或接口。
  2. 具体组件（Concrete Component）：实现了组件接口，是被装饰的对象。
  3. 装饰器（Decorator）：继承自组件类，并持有一个指向组件对象的引用，它定义了一个与组件接口一致的接口，并可以添加额外的功能。
  4. 具体装饰器（Concrete Decorator）：实现了装饰器接口，负责向被装饰的对象添加新的功能。

外观模式

外观模式（Facade Pattern）是一种结构型设计模式，它提供了一个统一的接口，用于访问子系统中的一组接口。外观模式通过将客户端与子系统的复杂性分离，为客户端提供了一个简化的接口，从而隐藏了子系统的复杂性。

外观模式就是给很多复杂的子系统提供一个简单的上层接口，并在这些接口中包含用户真正关心的功能。

外观模式主要包括以下几个角色：

  1. 外观（Facade）：为客户端提供了一个简化的接口，客户端通过外观接口与子系统进行交互，而不需要直接与子系统的组件进行交互。
  2. 子系统（Subsystem）：包含一组相互关联的类或接口，负责完成具体的功能。

享元模式

享元模式（Flyweight Pattern）是一种结构型设计模式，它旨在通过共享对象来最大程度地减少内存使用和提高性能。在享元模式中，共享的对象被设计成可共享的，并且可以在需要时被重复使用。

享元模式主要包括以下几个角色：

  1. 享元接口（Flyweight）：定义了享元对象的接口，通过这个接口可以接受并作用于外部状态。
  2. 具体享元（Concrete Flyweight）：实现了享元接口，并为内部状态（可共享状态）提供存储。具体享元对象必须是可共享的，可以在多个环境中重复使用。
  3. 非共享具体享元（Unshared Concrete Flyweight）：不可共享的享元对象，通常是因为其内部状态无法共享或者不适合共享的原因。
  4. 享元工厂（Flyweight Factory）：负责创建和管理享元对象，它维护一个享元池（Flyweight Pool），用于存储和管理已经创建的享元对象，确保可以重复使用。

代理模式

代理模式（Proxy Pattern）是一种结构型设计模式，它允许通过代理对象来控制对真实对象的访问。代理模式通常用于在访问一个对象时引入一定程度的间接性，从而可以在访问对象前后做一些额外的处理。

代理模式主要包括以下几个角色：

  1. 抽象主题（Subject）：定义了真实主题和代理主题的共同接口，客户端通过该接口访问真实主题和代理主题。
  2. 真实主题（Real Subject）：定义了代理所代表的真实对象。
  3. 代理主题（Proxy）：保存了对真实主题的引用，并提供了与真实主题相同的接口，客户端通过代理对象来间接地访问真实对象。

行为模式

责任链模式

责任链模式（Chain of Responsibility Pattern）是一种行为型设计模式，它允许多个对象在链式结构中依次处理请求，直到其中一个对象能够处理该请求为止。责任链模式将请求的发送者和接收者解耦，使得多个对象都有机会处理请求，从而提高了系统的灵活性和可扩展性。

在责任链模式中，通常会有一个抽象处理者（Handler）和具体处理者（Concrete Handler）。每个具体处理者都包含一个指向下一个处理者的引用，当收到请求时，它可以选择自己处理或者将请求传递给下一个处理者。

命令模式

命令模式（Command Pattern）是一种行为型设计模式，它允许将请求封装成对象，从而使得可以用不同的请求对客户端进行参数化，并且支持对请求排队、记录请求日志、撤销操作等。

在命令模式中，通常会有四个角色：

  1. 命令接口（Command）：声明了执行请求的方法 execute()。
  2. 具体命令（Concrete Command）：实现了命令接口，包含了对接收者的引用，并实现了具体的执行逻辑。
  3. 接收者（Receiver）：执行具体的操作。
  4. 调用者/请求者（Invoker/Client）：创建命令对象并设置其接收者，负责发出请求。

迭代器模式

迭代器模式（Iterator Pattern）是一种行为型设计模式，用于提供一种方法顺序访问一个聚合对象中的各个元素，而又不暴露其内部表示。迭代器模式将遍历元素的责任交给迭代器对象，而不是聚合对象本身，从而实现了解耦和更好的封装性。

在迭代器模式中，通常会有两个角色：

  1. 迭代器（Iterator）：定义了访问和遍历元素的接口，包括获取下一个元素、判断是否还有元素等方法。
  2. 具体迭代器（Concrete Iterator）：实现了迭代器接口，在具体聚合对象上进行迭代。

中介者模式

中介者模式（Mediator Pattern）是一种行为型设计模式，它用于减少对象之间的直接通信，而是通过一个中介对象来协调和控制各个对象之间的交互。这种模式提供了一种集中化的方式来管理复杂的关联关系，使得各个对象之间的耦合度降低，系统更易于维护和扩展。

在中介者模式中，通常会存在以下几个角色：

  1. 中介者（Mediator）： 定义了一个接口用于各个同事对象之间的通信，并负责协调各个同事对象的行为。
  2. 具体中介者（Concrete Mediator）： 实现了中介者接口，负责实际的协调工作，了解并维护各个同事对象之间的关系。
  3. 同事对象（Colleague）： 每个同事对象都知道中介者对象，但它们之间并不直接通信，而是通过中介者对象来进行消息的传递和交互。
  4. 具体同事对象（Concrete Colleague）： 实现了同事对象接口，负责自己的业务逻辑，并与其他同事对象通过中介者对象进行通信。

备忘录模式

备忘录模式（Memento Pattern）是一种行为型设计模式，用于捕获对象的内部状态，并在需要时将对象恢复到先前的状态。该模式通常用于需要实现撤销操作或者保存历史状态的场景。

角色说明：

  • 发起人（Originator）： 负责创建备忘录（Memento），并将自身状态保存到备忘录中或者从备忘录中恢复状态。
  • 备忘录（Memento）： 存储发起人对象的内部状态，在需要时可以将状态恢复到先前的状态。
  • 管理者（Caretaker）： 负责保存和管理备忘录对象，但不对备忘录的内容进行操作。

工作原理：

  1. 保存状态： 当发起人对象需要保存当前状态时，创建一个备忘录对象，并将当前状态保存到备忘录中。
  2. 恢复状态： 当需要恢复到先前的状态时，从备忘录中获取之前保存的状态，并将发起人对象恢复到该状态。

观察者模式

观察者模式（Observer Pattern）是一种行为型设计模式，用于定义对象之间的一对多依赖关系，当一个对象的状态发生变化时，所有依赖于它的对象都会得到通知并自动更新。这种模式也被称为发布-订阅（Publish-Subscribe）模式。

角色说明：

  • 主题（Subject）： 也称为被观察者或可观察对象，它维护一系列观察者对象，并提供注册和移除观察者的方法，以及通知观察者的方法。
  • 观察者（Observer）： 也称为订阅者或监听者，它定义了一个更新的接口，以便主题在状态发生变化时能够通知观察者进行更新。
  • 具体主题（ConcreteSubject）： 实现了主题接口的具体类，在状态发生变化时会通知所有注册的观察者。
  • 具体观察者（ConcreteObserver）： 实现了观察者接口的具体类，当收到主题的通知时会执行更新操作。

工作原理：

  1. 注册观察者： 观察者通过注册到主题上来订阅主题的状态变化。
  2. 状态变化通知： 主题在状态发生变化时会通知所有注册的观察者。
  3. 更新操作： 收到通知的观察者会执行相应的更新操作，以响应状态变化。

示例：

假设有一个天气数据的发布订阅系统，气象站（主题）会定期发布实时天气数据，而多个天气预报应用（观察者）可以订阅气象站的数据并进行展示。

在这个示例中，WeatherStation 类是具体主题类，WeatherObserver 类是具体观察者类。当气象站更新天气数据时，会调用 setTemperature() 方法来通知所有注册的观察者。观察者收到通知后，会调用 update() 方法来更新显示当前温度。7100

策略模式

策略模式（Strategy Pattern）是一种行为型设计模式，它定义了一系列算法，将每个算法封装到独立的类中，并使它们可以相互替换。策略模式使得算法可以独立于客户端而变化，从而使得客户端可以根据需要选择不同的算法。

角色说明：

  • 策略接口（Strategy）： 定义了所有支持的算法的通用接口。
  • 具体策略类（ConcreteStrategy）： 实现了策略接口，并封装了具体的算法。
  • 上下文（Context）： 维护一个对策略对象的引用，并负责将客户端的请求委托给具体的策略对象。

工作原理：

  1. 封装算法： 将每个算法封装到具体的策略类中。
  2. 选择算法： 在上下文中选择并保存一个具体的策略对象。
  3. 执行算法： 当客户端需要执行某个算法时，直接调用上下文对象的方法，并委托给当前选择的具体策略对象执行。

状态模式

状态模式（State Pattern）是一种行为型设计模式，它允许一个对象在其内部状态改变时改变其行为，使得对象看起来好像修改了其类。状态模式将对象的状态封装成独立的类，并将行为委托给当前状态对象。这样可以避免在对象内部使用大量的条件语句来切换行为，提高了代码的可维护性和可扩展性。

角色说明：

  • 环境类（Context）： 定义了客户感兴趣的接口，并维护一个具体状态子类的实例，用于定义当前的状态。
  • 抽象状态类（State）： 定义了一个接口，用于封装与环境对象的一个特定状态相关的行为。
  • 具体状态类（Concrete State）： 实现了抽象状态类定义的接口，并实现了与环境对象当前状态相关的行为。

工作原理：

  1. 定义状态： 将对象的各种状态封装成独立的状态类。
  2. 状态切换： 当对象的状态发生改变时，切换到对应的状态类。
  3. 委托行为： 将对象的行为委托给当前状态类处理。

示例：

假设有一个文档编辑器，可以根据文档的不同状态（未保存、已保存、已打开等）来执行不同的操作。我们可以使用状态模式来实现文档编辑器的状态管理。

在这个示例中，DocumentState 是抽象状态类，UnsavedState 和 SavedState 是具体的状态类，DocumentEditor 是上下文类。编辑器根据文档的状态执行不同的操作，例如保存、打开、关闭等。通过状态模式，可以轻松地管理文档编辑器的状态转换和行为。

模板方法模式

模板方法模式（Template Method Pattern）是一种行为型设计模式，它定义了一个操作中的算法的骨架，将一些步骤延迟到子类中。模板方法使得子类可以在不改变算法结构的情况下重新定义算法的某些步骤。

角色说明：

  • 抽象类（Abstract Class）： 定义了一个模板方法，其中包含了算法的框架和部分步骤的具体实现，这些步骤可以是抽象的，由子类去实现。
  • 具体类（Concrete Class）： 实现了抽象类中定义的抽象方法，完成算法中剩余的具体步骤。

工作原理：

  1. 定义模板方法： 在抽象类中定义一个模板方法，包含了算法的框架和一些具体步骤的实现。
  2. 实现具体步骤： 在具体类中实现抽象类中定义的抽象方法，完成算法的具体步骤。
  3. 执行模板方法： 调用抽象类中的模板方法，触发算法的执行，其中包含了具体类中实现的步骤。

示例：

假设有一个制作咖啡和茶的过程，其中煮水、冲泡和倒入杯中的步骤在两种饮料中都存在，但添加调料的步骤各不相同。我们可以使用模板方法模式来实现这个过程。

在这个示例中，Beverage 是抽象类，定义了一个模板方法 prepare()，其中包含了制作饮料的整个流程，包括煮水、冲泡、倒入杯中和添加调料。Coffee 和 Tea 是具体类，实现了 brew() 和 addCondiments() 方法，完成了咖啡和茶的具体制作过程。通过模板方法模式，我们可以在不改变制作饮料流程的情况下，方便地添加新的饮料类。

访问者模式

访问者模式（Visitor Pattern）是一种行为型设计模式，用于在不改变对象结构的前提下，为对象结构中的元素添加新的操作。它将操作封装成独立的访问者类，使得可以在不修改元素类的情况下定义新的操作。

角色说明：

  • 访问者（Visitor）： 定义了对每个元素的访问操作，可以访问不同类型的元素，并执行相应的操作。
  • 具体访问者（Concrete Visitor）： 实现了访问者定义的操作，对对象结构中的元素进行具体操作。
  • 元素（Element）： 定义了一个接受访问者的方法，接受一个访问者对象作为参数，以便让访问者访问自己。
  • 具体元素（Concrete Element）： 实现了元素定义的接受访问者方法，将自身传递给访问者，以便访问者对自己进行操作。
  • 对象结构（Object Structure）： 包含了一个元素的集合，可以是一个列表、数组、树等数据结构，提供了遍历元素的方法。

工作原理：

  1. 定义访问者接口： 确定访问者可以执行的操作，定义在访问者接口中。
  2. 实现具体访问者： 实现访问者接口中定义的操作，针对不同的元素进行不同的处理。
  3. 定义元素接口： 确定元素可以接受访问者的方法，定义在元素接口中。
  4. 实现具体元素： 实现元素接口中的接受访问者方法，将自身传递给访问者进行处理。
  5. 组织对象结构： 组织一个包含元素的集合，提供遍历元素的方法。
  6. 客户端调用： 创建具体访问者和具体元素对象，将具体元素对象添加到对象结构中，并调用访问者对象的访问方法。

示例：

假设有一个图形编辑器，其中包含了多种图形（如圆形、矩形、三角形等），我们需要为不同类型的图形添加不同的操作（如计算面积、计算周长等）。我们可以使用访问者模式来实现这个需求。

在上面的示例中，我们定义了图形编辑器中的几种图形（圆形、矩形、三角形）作为具体元素，以及一个面积计算器作为具体访问者。通过访问者模式，我们可以轻松地添加新的操作，例如计算周长、绘制图形等，而不需要修改图形类的代码，这符合开闭原则的设计理念。
